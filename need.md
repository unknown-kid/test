# AI辅助论文阅读Web系统 — 需求文档

## 1. 项目概述

本系统是一个面向科研人员的AI辅助论文阅读与管理平台。用户可以上传、管理和阅读学术论文（PDF格式），系统提供AI驱动的辅助阅读功能，包括智能对话、划词翻译、自动生成阅读报告等。系统支持多用户使用，具备完善的权限管理和文件管理能力。

### 1.1 预期规模

- 论文总量：约10万篇
- 用户数量：约100人
- 角色类型：系统管理员、普通用户

### 1.2 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端框架 | React |
| UI组件库 | Ant Design |
| PDF阅读器 | pdf.js (https://github.com/mozilla/pdf.js)|
| 后端框架 | FastAPI |
| 关系型数据库 | PostgreSQL |
| 向量数据库 | Milvus |
| 对象存储 | MinIO (S3兼容) |
| 异步任务队列 | Celery + Redis |
| PDF解析 | PyMuPDF + pdfplumber |
| 多智能体框架 | CrewAI（用于阅读报告生成） |
| 认证方式 | JWT Token |
| 部署方式 | Docker Compose |
| 界面语言 | 仅中文 |

## 2. 用户角色与权限

### 2.1 角色定义

#### 2.1.1 系统管理员

- 通过独立的管理员后台路由（同一React应用内，路由区分）登录
- 管理共享论文区（上传、删除、移动、复制、新建文件夹等全部文件操作）
- 不能查看或操作普通用户的个人区
- 配置系统超参数（详见第7节）
- 配置翻译模型、嵌入模型、默认对话模型接口
- 管理普通用户（审批注册、移除用户、查看用户信息看板）
- 移除用户时需二次确认，并提示该操作将直接删除该用户的所有相关数据（个人区论文、对话记录、高亮、批注、笔记、个人阅读报告等，含MinIO文件和Milvus向量数据）
- 不提供论文详情页（管理员后台不需要阅读论文）
- 预置管理员账号：admin/admin123，管理员可在后台修改自己的密码

#### 2.1.2 普通用户

- 通过前端页面登录
- 对个人区拥有全部文件操作权限
- 对共享论文区仅有查看、阅读、复制到个人区（完整复制）的权限
- 可在论文详情页进行AI辅助阅读
- 可自行配置个人对话模型接口（覆盖管理员默认配置，仅对自己有效）
- 可清除自定义对话模型配置，回退到使用管理员默认模型
- 用户配置对话模型时提供"测试连接"按钮验证API有效性

### 2.2 用户注册与认证

- 注册方式：用户自行注册，管理员审批通过后方可使用
- 认证方式：JWT Token，前后端分离
- JWT Token过期时间默认1天（管理员可配置）
- 采用双Token机制：短期access token + 长期refresh token，支持无感刷新
- 管理员与普通用户使用不同的登录入口（同一应用，不同路由）

## 3. 文件管理系统

### 3.1 整体结构

文件系统分为两大区域：

```
根目录/
├── 共享论文区/          # 管理员管理，普通用户只读
│   ├── 文件夹A/
│   │   ├── 论文1.pdf
│   │   └── 论文2.pdf
│   └── 文件夹B/
└── 用户个人区/          # 每个用户独立隔离
    ├── 用户A/
    │   ├── 文件夹X/
    │   └── 论文3.pdf
    └── 用户B/
        └── ...
```

### 3.2 权限矩阵

| 操作 | 管理员-共享区 | 普通用户-共享区 | 普通用户-自己个人区 | 普通用户-他人个人区 |
|------|:---:|:---:|:---:|:---:|
| 查看/浏览 | ✅ | ✅ | ✅ | ❌ |
| 阅读论文 | ❌（无详情页） | ✅ | ✅ | ❌ |
| 上传 | ✅ | ❌ | ✅ | ❌ |
| 删除 | ✅ | ❌ | ✅ | ❌ |
| 移动 | ✅ | ❌ | ✅ | ❌ |
| 复制 | ✅ | 仅完整复制到个人区 | ✅ | ❌ |
| 新建文件夹 | ✅ | ❌ | ✅ | ❌ |
| 搜索 | ✅ | ✅ | ✅ | ❌ |

注：管理员不能访问或操作任何用户的个人区。用户不能将个人区论文移动到共享区。

### 3.3 文件操作功能

- 支持批量和单次操作：上传、删除、移动、复制、新建文件夹
- 文件夹本身也支持移动和复制操作（类似Windows文件管理器）
- 同区域内的复制为深拷贝（复制PDF文件、生成新paper_id、复制所有论文元数据和处理结果，不复制用户个人数据）
- 深拷贝通过Celery异步任务执行，不重新跑5个处理步骤，直接复制已有结果（含Milvus向量数据以新paper_id重新插入）
- 深拷贝失败时全量回滚（删除已复制的部分数据），仅保留所有信息都复制完成的论文
- 跨区复制（共享区→个人区）为完整复制（深拷贝），复制PDF文件、生成新paper_id、复制所有论文元数据和5个处理步骤的结果数据（含Milvus向量数据以新paper_id重新插入），不复制其他用户的个人数据
- 跨区深拷贝同样通过Celery异步任务执行，失败时全量回滚
- 移动/复制时的冲突处理：
  - 同名文件夹：弹出询问窗口，提供"全部覆盖"和"全部合并"两个选项
  - 同名文件：弹出询问窗口，提供"全部覆盖"和"全部跳过"两个选项
  - 冲突询问窗口串行弹出（先处理文件夹冲突，再处理文件冲突）
- 支持全选功能
- 每个文件夹显示其包含的论文总数量（递归统计所有子目录，直到全部叶子节点），通过folders表中的缓存计数字段维护，在论文增删移动时自动更新
- 文件夹嵌套深度上限：10层
- 不提供回收站功能
- 论文展示形式：类似文件管理器的图标+文件名，支持下拉列表形式展示
- 每篇论文展示信息：图标、文件名、状态标签、文件大小、上传时间
- 文件列表支持虚拟滚动+分页，分页可选10/20/50/100篇，默认每页20篇

### 3.4 搜索功能

搜索功能集成在文件系统页面中（非独立页面），包含两种搜索模式，可单独使用也可组合使用。默认搜索当前浏览的目录及其所有子目录。搜索结果的展示方式与文件系统页面一致。用户可在共享区和个人区的任意目录下进行搜索。

#### 3.4.1 关键词搜索

- 搜索范围：当前文件夹目录及其所有子目录，不搜索父目录
- 支持正则表达式匹配
- 支持多关键词搜索：关键词之间用英文分号 `;` 分隔
- 多关键词搜索逻辑：所有搜索关键词必须全部出现在"标题+关键词列表"中（标题视为关键词列表的一员），使用正则匹配（非全匹配，如搜索"test"可命中"testing"或"retested"），关键词之间为AND逻辑
- 搜索结果按文件顺序展示

#### 3.4.2 RAG语义搜索

- 用户可输入自然语言检索语句
- 搜索范围：用户选定的文件夹目录及其所有子目录
- 系统通过向量检索从论文的摘要向量中进行语义匹配
- 返回所有相似度大于阈值（threshold）的论文，阈值默认0.5，管理员可配置
- 返回语义相关的论文列表，按相似度降序排列，展示方式与文件系统页面一致

#### 3.4.3 组合搜索（两种模式可级联使用）

支持两种级联方向：

- 关键词搜索 → RAG搜索：先通过关键词搜索缩小范围，再在搜索结果集内输入RAG检索语句进行语义精筛
- RAG搜索 → 关键词搜索：先通过RAG语义搜索获取初步结果，再在结果集内使用关键词搜索进行精确过滤

两种级联方向均在文件系统页面的搜索区域中通过交互操作完成，用户可自由选择先后顺序。两个搜索方式的输入框应该分开，因为内容不同


### 3.5 论文唯一标识

- 每篇论文拥有唯一的论文ID（系统生成）
- 论文ID用于关联所有交互数据（对话记录、高亮、批注等）

## 4. 论文上传自动化处理

论文上传后自动执行以下5个步骤，所有步骤通过Celery异步任务队列执行。5个步骤互相独立，可全部并行执行（分块向量化、标题提取、摘要提取、关键词提取、阅读报告生成均可并行）。阅读报告不依赖前四步的任何数据。

在执行5个步骤之前，系统先使用PyMuPDF + pdfplumber提取论文全文文本。若提取结果为空，则判定为扫描版PDF，通过WebSocket实时推送通知用户该PDF为扫描版不支持处理，并自动删除该PDF的所有已入库数据，终止后续处理。

### 4.1 并发控制

- 批量上传时，对同一个配置的模型（LLM、嵌入模型、翻译模型）的并行请求量上限为64个（管理员可配置）
- 超过上限的请求需排队等待，不可超限并发
- 论文级并发控制：同时最多处理10篇论文的异步任务（批量上传和一键补全均受此限制）

### 4.2 步骤一：论文分块与向量化

- 使用PyMuPDF + pdfplumber提取论文全文文本
- 对全文进行分块处理，每个chunk大小默认3000字符（管理员可配置）
- 分块冗余策略：连续块之间有20%的重叠内容（重叠比例管理员可配置），保证关键数据不被截断
- 对每个分块进行向量化（使用管理员配置的嵌入模型）
- 向量数据存储到Milvus

### 4.3 步骤二：论文标题提取与去重

提取流程（串行，逐级降级）：

1. 使用PyMuPDF提取PDF元数据中的标题
2. 若步骤1未提取到，使用pdfplumber提取PDF元数据中的标题
3. 若步骤2仍未提取到，使用LLM从论文前N个字符中提取标题（N默认2000，管理员可配置）
4. 若以上均失败，使用原始上传文件名作为标题兜底

去重逻辑：

- 提取到标题后，检查同目录下是否存在同名论文（不同目录不视为重复）
- 批量上传时，去重在本批次所有论文的标题提取全部完成后统一进行，避免并发竞争导致的重复遗漏
- 若同名且文件大小一致：判定为重复论文，自动忽略新上传的论文，并通过WebSocket实时推送通知用户"检测到重复论文，已跳过"
- 若同名但文件大小不同：视为不同论文，两者均保留（文件名添加区分后缀）

最终提取到的标题作为该论文在系统中的显示名称。

### 4.4 步骤三：论文摘要提取

- 使用基于LLM的Agent
- 从论文前N个字符中提取摘要（N默认10000，管理员可配置）
- 提取完成后自动对摘要进行向量化
- 向量化数据存储到Milvus
- 原始数据和向量化数据要与论文id关联

### 4.5 步骤四：论文关键词提取

- 使用基于LLM的Agent
- 从论文前N个字符中提取关键词（N默认30000，管理员可配置）
- 提取关键词数量：M个（M默认20，管理员可配置）
- 关键词列表要与论文id关联

### 4.6 步骤五：自动生成论文阅读报告

使用基于CrewAI的多智能体架构，分两阶段生成：

CrewAI使用的LLM模型选择逻辑：
- 共享区论文（系统自动生成）：使用管理员配置的默认对话模型
- 个人区论文（用户生成/一键补全）：优先使用用户自定义对话模型，未配置则使用管理员默认模型

#### 阶段一：粗读（Outline Agent）

- 输入：论文前N个字符（N默认100000，管理员可配置）；若为用户生成的报告，还包含用户提供的关注点
- 输出：阅读报告的大纲（outline）

#### 阶段二：精读（Detail Agent）

- 输入：完整论文文本（N默认100000，管理员可配置） + 阶段一生成的大纲；若为用户生成的报告，还包含用户提供的关注点
- 输出：完整的论文阅读报告（Markdown格式）

- 论文阅读报告要与论文id关联

### 4.7 处理状态可视化

- 每篇论文的5个自动化步骤需要有可视化进度展示
- 文件系统页面中每篇论文旁显示总体状态标签（已完成/处理中/失败），鼠标悬停时显示各步骤详细状态
- 管理员在共享区查看论文时显示完整的处理状态
- 展示每个步骤的状态：等待中 / 处理中 / 已完成 / 失败
- 全部完成后，论文打上"已完成"标签
- 任一步骤失败，论文打上"失败"标签
- 文件系统页面提供"一键补全"功能：扫描当前访问目录及其所有子目录中的论文，自动识别未完成全部5个步骤的论文，对未完成的步骤重新执行（受论文级并发控制限制，同时最多处理10篇）
- 一键补全触发后前端显示进度条（已处理X/总共Y篇），不逐篇通知
- 一键补全权限：用户仅可在个人区目录下触发，不可在共享区触发；管理员可在共享区触发

## 5. 论文详情页

打开论文时，系统以新浏览器标签页进入该论文的详情页。新标签页通过localStorage共享JWT Token实现免登录认证（同域同源SPA自动共享）。

### 5.0 共享区与个人区论文阅读区别

- 共享区论文：用户仅可查看PDF内容（使用pdf.js渲染器），不可使用高亮、批注、问AI、翻译等交互功能。右侧为单一信息面板（非标签页形式），合并显示论文详细信息（不含处理状态）和系统阅读报告（若报告尚未生成完成则显示"报告生成中"状态）。页面右上角提供"复制到个人区"按钮，点击后展示个人区文件夹结构供用户选择目标文件夹，触发异步深拷贝，完成后通过WebSocket通知用户并自动跳转到个人区文件系统页。
- 个人区论文：用户可使用全部功能（高亮、批注、翻译、问AI、AI对话、个人笔记、个人阅读报告等）。

### 5.1 PDF阅读器（左侧主区域）

基于pdf.js实现，需支持以下功能：

- PDF文件通过后端API中转加载（非直接访问MinIO），后端根据paper_id返回PDF流
- 高亮和批注数据由后端按pdf.js格式组装后返回前端

#### 基础功能

- 页面缩放（放大、缩小、适应宽度、适应页面）
- 页面滚动浏览
- 页码跳转
- 键盘快捷键支持（常规PDF阅读快捷键）
- 划词复制

#### 划词交互功能

划词后弹出统一的选项框（pdf.js原生功能与自定义功能合并在同一弹出框中），包含以下功能选项：

| 功能 | 说明 |
|------|------|
| 复制 | 复制选中文本到剪贴板 |
| 高亮 | 对选中文本添加高亮标记，持久化保存（复用pdf.js原生高亮功能） |
| 批注 | 对选中文本添加文字批注，持久化保存（复用pdf.js原生批注功能） |
| 翻译 | 划词翻译功能（详见5.1.1） |
| 问AI | 划词问AI功能（详见5.1.2） |

高亮和批注数据按照不同的用户进行隔离，相互之间不可见

注：高亮和批注功能完全复用pdf.js原生实现，不做自定义增量开发。划词弹出框中的"翻译"和"问AI"为自定义实现的功能。

#### 5.1.1 划词翻译

- 划词后选择"翻译"，弹出独立对话框
- 对话框中显示原文和翻译结果
- 翻译目标语言固定为中文
- 只有管理员才能配置的翻译模型接口进行翻译

#### 5.1.2 划词问AI

- 划词后选择"问AI"
- 将选中文本作为上下文
- 通过RAG从该论文的向量库中检索相关内容
- 使用LLM对检索结果进行Rerank，取排名前K个数据块（K默认10，管理员可配置）
- Rerank使用当前用户的对话模型（用户配置了用用户的，否则用管理员默认配置的）
- 将划词选中文本 + Rerank后的数据块一起作为上下文
- 自动在右侧AI对话栏创建一个新的对话session，用户在其中继续提问

### 5.2 右侧交互窗口

右侧窗口包含四个导航栏标签页：

#### 5.2.1 AI对话

- 支持与当前论文相关的AI问答对话
- 支持流式输出（SSE），对输出内容使用Markdown渲染器进行结果渲染
- 一篇论文下用户可创建多个对话session，数量无上限
- session列表以抽屉推拉方式展示（可拉出查看，可收起隐藏）
- session支持删除（需确认弹窗），不支持重命名
- session标题自动取用户第一条消息的前N个字符（N默认30，管理员可配置）
- 划词问AI创建的session在列表中有特殊标识，区分于普通对话session
- 每次对话的上下文构成：
  - 该论文的前P个数据块（P默认5，管理员可配置），按chunk_index顺序取前P个
  - 通过向量检索 + LLM Rerank后的前Q个数据块（Q默认5，管理员可配置）
  - 每轮对话都重新进行向量检索和Rerank（以用户最新提问作为查询，重新检索并拼接上下文）
- 上下文截断策略：
  - 上下文窗口上限默认100000字符（管理员可配置）
  - 优先保留：用户最新提问 + 完整对话历史（不限轮数，只要总字符不超上限）
  - 可截断：检索出的上下文数据块（当总上下文超限时，从Rerank排名靠后的chunk开始截断）
  - 当对话历史超过60000字符时，提醒用户当前session已接近上下文上限，建议新建session
- Rerank使用当前用户的对话模型（用户配置了用用户的，否则用管理员默认配置的）
- 可选功能：用户可在论文详情页全局设置是否将论文阅读报告加入对话上下文
  - 若开启，划词问AI（5.1.2）和普通AI对话的上下文中均额外包含阅读报告
  - 优先使用用户个人阅读报告，若无个人报告则使用系统生成的报告
- 历史记录功能：
  - 保存完整的对话历史
  - 记录每条消息的时间戳
  - 前端展示每条历史记录的最后更新时间
  - 历史记录按论文维度隔离，每篇论文独立的对话历史
- 所有聊天记录在不同用户之间进行隔离

#### 5.2.2 AI阅读报告

- 展示自动生成的论文阅读报告（步骤五的产出），报告内容为Markdown格式，前端使用Markdown渲染器展示
- 如果报告尚未生成完成，显示生成进度/状态
- AI阅读报告栏可以选择查看现在已有的阅读报告，以下拉菜单形式选择
- 每个用户仅可以在AI阅读报告栏生成一个自己的阅读报告，这个阅读报告与系统自动生成的报告为两个分开的报告，用户不能对系统生成的报告进行任何操作，但是可以删除自己生成的报告，并且重新生成（提供一个重新生成按钮）。重新生成时保留旧报告直到新报告生成完成后替换，生成失败则保留旧报告不变。用户自己的阅读报告生成过程如下：

使用基于CrewAI的多智能体架构，分两阶段生成（模型选择：优先使用用户自定义对话模型，未配置则使用管理员默认模型）：

#### 阶段一：粗读（Outline Agent）

- 输入：论文前N个字符（N默认100000，管理员可配置）+ 用户上传的论文阅读关注点（用户点击重新生成时提供对话框要求用户提供相关关注点文本描述）
- 输出：阅读报告的大纲（outline）

#### 阶段二：精读（Detail Agent）

- 输入：完整论文文本（N默认100000，管理员可配置） + 阶段一生成的大纲 + 用户关注点
- 输出：完整的论文阅读报告（Markdown格式）

- 论文阅读报告要与论文id关联
- 用户生成的阅读报告在不同用户之间进行隔离

#### 5.2.3 论文详细信息

- 展示论文的元数据信息：
  - 论文标题
  - 论文摘要
  - 论文关键词
  - 文件大小
  - 上传时间
  - 处理状态

#### 5.2.4 个人笔记

- 用户可以针对当前论文撰写个人笔记
- 笔记支持纯文本和Markdown格式编辑
- 笔记内容持久化保存
- 按用户隔离，每个用户只能看到自己的笔记

### 5.3 用户数据隔离

以下数据按用户隔离，每个用户只能看到自己的数据：

- AI对话记录（含时间戳）
- 划词高亮记录
- 划词批注记录
- 个人笔记
- 个人版AI阅读报告

用户打开论文时，自动加载自己之前的所有记录。

## 9. 非功能性需求

### 9.1 性能要求

- 支持10万篇论文的存储和检索
- 支持100个并发用户
- 文件系统浏览响应时间 < 1秒
- 关键词搜索响应时间 < 5秒（10万篇论文规模）
- 向量检索响应时间 < 3秒
- PDF加载时间 < 5秒（常规大小论文）

### 9.2 数据存储架构

- 论文PDF文件：MinIO对象存储
- 论文元数据、用户数据、交互数据、文件系统结构：PostgreSQL
  - 关键词搜索使用pg_trgm扩展建立全文索引加速正则匹配
- 论文分块向量、摘要向量：Milvus
  - 向量检索时支持paper_id列表过滤，限定搜索范围到指定目录下的论文
- 异步任务队列：Redis（作为Celery Broker）
- 会话缓存：Redis

### 9.3 安全要求

- JWT Token认证，双Token机制（access token + refresh token），支持无感刷新
- 用户数据严格隔离
- API接口权限校验
- API Key等敏感信息加密存储
- 文件上传大小限制：单个PDF文件最大100MB
- 文件类型校验（仅允许PDF）
- 扫描版PDF（图片型PDF）检测：使用PyMuPDF + pdfplumber提取文本，若提取结果为空则判定为扫描版PDF，通过WebSocket实时推送通知用户该PDF为扫描版不支持处理，并自动删除该PDF的所有已入库数据
- 论文删除需事务保证：PostgreSQL记录、MinIO文件、Milvus向量数据三者的删除操作需保证一致性，任一存储删除失败则提示用户未成功删除

### 9.4 实时通知

- 每个页面（文件系统页、论文详情页等）建立独立的WebSocket连接，采用基于用户维度的连接管理：同一用户的多个页面共享同一逻辑通知通道，服务端按user_id投递消息，各页面连接均可收到
- WebSocket连接使用access token进行初始认证，连接建立后服务端维护会话状态，不因token过期断开
- WebSocket断线自动重连：网络断开后客户端自动重连，重连时使用当前有效的access token（必要时先通过refresh token刷新）
- 维护模式下：服务端先通过WebSocket推送维护通知，然后主动关闭所有普通用户的WebSocket连接，再清空token
- 前端提供通知按钮，用户可点击查看通知消息列表
- 通知持久化存储：用户不在线时的通知保存到数据库，上线后加载未读通知
- 通知状态：支持已读/未读状态标记，用户可手动标记已读
- 通知自动清理：通知创建时间超过3天后自动删除（通过Celery Beat定时任务实现），无论已读未读均会清理
- 用户可手动删除任意通知
- 系统通过WebSocket向用户实时推送通知，包括：
  - 异步任务状态变更（上传处理完成/失败、深拷贝完成/失败）
  - 去重检测到重复论文
  - 扫描版PDF不支持处理
  - 深拷贝空间不足（清理未完成的部分数据，通知用户）
  - 系统维护模式开启/关闭通知
- 系统需保证能存放10万篇论文及其相关信息的存储容量

### 9.5 系统维护模式

- 管理员后台提供"系统维护模式"开关按钮
- 开启维护模式后：普通用户无法登录，已登录用户被强制登出（服务端先通过WebSocket推送维护通知，然后主动关闭所有普通用户的WebSocket连接，再清空Redis中所有普通用户的refresh token），仅管理员可登录
- 维护模式下管理员仅可执行：全量向量重建、模型配置，不可执行其他任何操作
- 维护模式用于嵌入模型更换后的全量向量重建等需要系统独占的操作，仅在更换嵌入模型时触发，更换翻译模型或对话模型不触发维护模式
- 开启/关闭维护模式时通过WebSocket通知所有在线用户

### 9.6 部署架构

使用Docker Compose编排以下服务：

- 前端服务（React + Nginx）
- 后端服务（FastAPI + Uvicorn）
- PostgreSQL
- Milvus
- MinIO
- Redis
- Celery Worker（可水平扩展）
